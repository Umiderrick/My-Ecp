第一章绪论
1.1引言
   未来必然是移动互联网的天下，商品在工厂、在仓库、在货架、在虚拟的线上线下都不要紧，通过互联网的技术连接起来，再通过网络上各自的“通道”，直达消费者。电子商务的定义不只是淘宝京东式的电商。归根结底是通过信息流直通消费者。
1.2研究背景
    电子商务并不是新兴的事物，而是伴随着互联网技术和社会发展、业态的改变，消费者在互联网平台拥有更大的话语权，在与商家的博弈中，逐步占据主要地位，也就促使了消费市场由原来的以企业为中心过渡到以用户需求为中心的新趋势，由原来的集团集权商业体系转变到民主的商业体系，也就自然地推动传统的实体商业模式，必须适应以互联网消费者为中心的延伸及产业的正常升级。商家必须正视消费者的消费习惯的转移，正视消费者在消费端已经主导话语权，必须转变传统企业本位的消费观念才能够获得新业态下的竞争优势。这正是当今世界企业面临的电商转型背景。

当前，我国电子商务发展正处于密集创新和快速扩的新阶段，日益成为拉动我国消费需求、促进传统产业升级、发展现代服务业的重要引擎。具体而言，具有以下几个特点。

第二章电商网站及运营的基础知识
2.1 电商概念及作用
电商网站就是企业、机构或者个人在互联网上建立的一个站点，是企业、机构或者个人开展电商的基础设施和信息平台，是实施电商的交互窗口，是从事电商的一种手段。
作用：
1.是进行电子商务活动信息交流的枢纽
2.是企业商家的门户
3.是企业开展网络营销的工具
4.是体现电商个性化服务的平台
我想对于大学生应该观察到社会中的机会，电商虽然热潮依旧，但是和传统的门户网站相比。新兴的电商很难被别人接受，大家的盈利模式都很相似，没有创新就很难活过初期阶段。通过自己实现一个简易的平台可以提升对现有平台的许多方面理解，也可以实际上锻炼自己的技术和思维。
2.1.1运维过程
电子商务运营与企业运营存在相似之处，包括调研 、产品定位、管理分类、开发规划、运营策划、产品管控、数据分析、分析执行及跟进等。但其执行对象有别于实体产品。电子商务运营的对象是根据企业需要所开发设计建设的电子商务平台的附属宣传推广产品
电商平台的搭建其实相比运维简单很多，经营一个B2C电商平台主要以：网络零售业为主，主要借助于Internet开展在线销售活动。需要不同的职能人员：库存管理的，平台界面的美工，每个商品上架的详情页也需要专业的人操作，在用户下单之后又应该进行订单管理（如删除重复订单和订单中数量更改的操作），在相关操作完成后最后还有的配送阶段就暂时未进入设计层面。
2.1.2盈利模式
电商网站主要以销售商品为主要盈利手段。刚开始起步只会以自营的商品为销售对象经过一段时间的盈利之后，或者经过用户群体的逐步扩大，会积累一部分用户，当有传统品牌发现这个网站相比其他店铺租金更低廉，销售量更多，就会将自己的品牌放入该网站进行推广和销售这就是后期盈利的方式。用户量进一步庞大或者品牌已经获得行内的一致好评就会有广告上投放广告，为更多的商品提供线上消费的渠道。那是网站的盈利基本就可以保持增长，寻找更大的盈利空间和网站的改革。

随着访问量继续增加，单台应用服务器已经无法满足需求了。在假设数据库服务器没有压力的情况下，我们可以把应用服务器从一台变成了两台甚至多 台，把用户的请求分散到不同的服务器中，从而提高负载能力。多台应用服务器之间没有直接的交互，他们都是依赖数据库各自对外提供服务。著名的做故障切换的 软件有keepalived，keepalived是一个类似于layer3、4、7交换机制的软件，他不是某个具体软件故障切换的专属品，而是可以适用 于各种软件的一款产品。keepalived配合上ipvsadm又可以做负载均衡，可谓是神器。
　　我们以增加了一台应用服务器为例，增加后的系统结构图如下：

　　系统演变到这里，将会出现下面四个问题：
1.用户的请求由谁来转发到到具体的应用服务器
2.有什么转发的算法
3.应用服务器如何返回用户的请求
4.用户如果每次访问到的服务器不一样，那么如何维护session的一致性
　　我们来看看解决问题的方案：
第一个问题即是负载均衡的问题，一般有5种解决方案：
　　　　1、http重定向。HTTP重定向就是应用层的请求转发。用户的请求其实已经到了HTTP重定向负载均衡服务器，服务器根据算法要求用户重定向，用户收到重定向请求后，再次请求真正的集群 优点：简单。缺点：性能较差。
　　　　2、DNS域名解析负载均衡。DNS域名解析负载均衡就是在用户请求DNS服务器，获取域名对应的IP地址时，DNS服务器直接给出负载均衡后的服务器IP。优点：交给DNS，不用我们去维护负载均衡服务器。缺点：当一个应用服务器挂了，不能及时通知DNS，而且DNS负载均衡的控制权在域名服务商那里，网站无法做更多的改善和更强大的管理。
　　　　3、反向代理服务器。在用户的请求到达反向代理服务器时（已经到达网站机房），由反向代理服务器根据算法转发到具体的服务器。常用的apache，nginx都可以充当反向代理服务器。优点：部署简单。缺点：代理服务器可能成为性能的瓶颈，特别是一次上传大文件。
　　　　4、IP层负载均衡。在请求到达负载均衡器后，负载均衡器通过修改请求的目的IP地址，从而实现请求的转发，做到负载均衡。优点：性能更好。缺点：负载均衡器的宽带成为瓶颈。
　　　　5、数据链路层负载均衡。在请求到达负载均衡器后，负载均衡器通过修改请求的mac地址，从而做到负载均衡，与IP负载均衡不一样的是，当请求访问完服务器之后，直接返回客户。而无需再经过负载均衡器。

1.显示产品分类信息：即查询操作拼出sql“select  *  from  firtype”即可查询出所有一级类别，并给予显示第一类别名称。之后查询第二章表sectype讲查询的结果按外键-firtypeid分组并展示即可形成二级类别在第一类别下的显示，同理可以将查询三级类别的表展示。
2.添加产品分类信息：产品的分类的插入分三级显示:第一类别的插入不需要检查，只需要输入主键和名称即可录入。二级类别由于有外键干扰，需要先选择外键再选择自己的主键（为方便理解可将前几位设置为第一类别的id）在确定完主键之后即可输入二级分类的名称。三级分类只需二级分类id和自己的类别名和id不需要一级类别id由于之前已经有约束即可不做判断。
3.修改产品分类信息：
每个类别（三张表）可以通过自己的主键来管理每一个类别sql “update  from  表 set  列名 =‘新的值’  where  primary key = ‘主键值’ ”改功能暂时比较简陋不能完成批量操作可能对后期目标有更多要求。
4.删除产品分类信息:
同更改的原理由于一个表中每个类别只有一个主键删除的话也仅仅按主键删除即可，不过要注意删除母表的类别之前，先得把字表中与母表主键即子表外键包含的所有列名都删除否则会引起数据库报错。该操作会失败并弹出异常。


4.3 商品操作
1.商品录入功能：
商品在展示是为保证一致性首先就得用一个主键囊括所有自己包含的信息如必须要素：余量、价格等等。为保证表结构可扩展，暂定两个必有属性。其余不同商品对应属性不一样所面临的是根据外键加载剩余的属性如：厂商、型号。。还有一种必须的检索方式及根据类别查找目前只涉及了三级类别的查找剩余的方式可以通过表连接来扩展暂时没有涉及。

2.商品修改功能：
功能与类别修改相似、同是以主键来更新各个列名。

3.商品删除功能：
与类别删除相似传入主键，需注意删除前要确定该主键绑定的商品属性已经不存在否则会以异常形式抛出使删除功能失败。

4.商品图片录入功能：
商品显示不能仅以id方式还需要以至少一张图片来展示商品的信息。所以增加了图片录入功能及在数据库表明中增加一列叫imgurl 负责存放商品图片的相对路径。给表单设计enctype =”multipart/form-data”并使用post方法提交。服务器端使用request.getParamter方法获取参数值。此时调用getInputStream方法获得字节流，并分析流得出数据。我使用的工具是apache提供的：common-fileupload-1.2.1.jar。

5.生成缩略图与放大功能：
设计放大功能仅限于前端页面展示当鼠标移动到展示图片上，会生成一个新的图层紧邻原原图像选择光标区域：100px*100px放大为400px*400px的图像方便会员查看细节。
6.商品分类查询页面功能：
该功能通过使用sql join语句将表item_att和attribute 连接成一个新的表给予展示，再将attrig表中各项查询出来再用之前查出的结果筛选和展示将item和属性字典关联方便下单结果显示商品属性。
7.商品搜索查询功能：
根据sql”select * from item where itemname like  ‘%匹配字符%’”查出item包含的信息并在商品查询页面封装成list并显示结果  6
8.商品详情页面显示：
商品详情功能包含商品属性主子表：商品图片显示，商品余量与价格。还显示了评价页面包含信息：评价内容、评价人姓名。
